Security / Standard & Extended ACL / In & Out

「全部通す」から「選んで通す」へ。

ここまでの単元は、通信が「届くようにする」ための学習でした。この単元では ACL(アクセスコントロールリスト)を使って、通信を意図的に許可・拒否します。番号で指定するタイプと名前で指定するタイプの両方を実際に作り、止まる瞬間・止まらない瞬間をコマンドで観察します。特に混同しやすい in と out の違いは、わざと間違った方向に適用する実験まで行って、確実に身につけます。

Why ACL

ACLとは何か: 上から順に照合される「通行ルールのリスト」

ACL は「どんなパケットを(条件)、通すか止めるか(permit / deny)」を1行ずつ並べたリストです。Router はパケットを受け取るたびにリストを上から順に照合し、最初に一致した行の permit / deny で処理を即決します。そして、どの行にも一致しなかったパケットは、リストの末尾に隠れている「暗黙の deny」によってすべて破棄されます。

標準 ACL(番号 1〜99)

  • 判断材料は送信元 IP アドレスだけです
  • 「誰から来たか」しか見ないため、宛先を区別できません
  • 送信元の近くで止めると、無関係な宛先への通信まで巻き添えにするので、宛先の近くに置くのが定石です

拡張 ACL(番号 100〜199)

  • 送信元・宛先・プロトコル・ポート番号まで細かく指定できます
  • 「PC-A から Server への HTTP だけ拒否」のような狙い撃ちができます
  • 条件が細かく巻き添えの心配がないため、無駄なパケットを早く捨てられる送信元の近くに置くのが定石です

「番号で指定」と「名前で指定」の2つの書き方

標準・拡張のどちらの ACL も、access-list 10 ... のように番号で作る書き方と、ip access-list extended BLOCK-A-WEB のように名前で作る書き方があります。名前付きは「何のためのルールか」が一目でわかるうえ、行単位での削除・挿入ができるという実務上の大きな利点があります。この単元では、番号付きの標準 ACL と、名前付きの拡張 ACL を1つずつ作って両方を体験します。

Server の HTTP を使う理由

Packet Tracer の Server は、Services タブで HTTP が最初から有効になっています。この単元では Ping(ICMP)と Web(HTTP)という2種類の通信を用意し、後半の拡張 ACL で「Ping は通るのに Web だけ止まる」という選別を体験します。

Packet Tracer で選ぶ Router は 1941

この単元のコマンド例は、GigabitEthernet ポート(G0/0・G0/1)を持つ 1941(2911 でも可)を選んだ前提です。1841 のような FastEthernet 搭載機種を選んだ場合は、interface gigabitEthernet0/0 を interface fastEthernet0/0 のようにポート名を読み替えてください。ACL は「どのポートのどの方向に付けるか」がすべてなので、ポート名の取り違えは結果に直結します。実際のポート名は show ip interface brief で確認できます。詳しくはPacket Tracer の使い方の「装置の選び方とポート名の関係」を参照してください。

In or Out

最重要: in と out は「Router を主人公」にして考える

ACL をインターフェースに適用するときは、必ず in か out の方向を指定します。ここで PC 視点(送る側・受ける側)で考えると必ず混乱します。主人公は Router です。そのインターフェースから Router に「入ってくる」パケットを検査するのが in、Router から「出ていく」パケットを検査するのが out です。

in = Router に入ってくる瞬間に検査

G0/0 に in で適用すると、LAN 側から R1 に飛び込んでくるパケットが検査対象になります。合格したパケットだけが R1 の中に入り、経路表に従って転送されます。不合格なら、転送の処理に入る前に捨てられます。

out = Router から出ていく瞬間に検査

G0/1 に out で適用すると、R1 が Server 側へ送り出す直前のパケットが検査対象になります。どこから入ってきたかは関係なく、「このドアから外へ出ようとするパケット」がすべて検査されます。

Ping 1往復で通過する検査ポイントは4箇所

PC-A から Server へ Ping を打つと、パケットは次の4つの検査ポイントを順に通過します。ACL はこのどこにでも置けます。

  • 行き(送信元 PC-A → 宛先 Server): G0/0 の in → 経路判断 → G0/1 の out
  • 帰り(送信元 Server → 宛先 PC-A): G0/1 の in → 経路判断 → G0/0 の out

ここで重要なのは、行きと帰りでは送信元と宛先が入れ替わることです。「PC-A が送信元のパケット」は G0/0 の in と G0/1 の out にしか現れません。帰りのパケットの送信元は Server だからです。この事実が、後のハンズオンの実験で効いてきます。迷ったら「自分がパケットになって、いま R1 に入る瞬間なのか、出る瞬間なのか」を想像してください。

Step by Step

ハンズオン: 作る → 当てる → わざと間違える

STEP 1

トポロジを組んで「全部通る」状態を確認する

機器を配置してケーブルをつなぎ、R1 に IP を設定します。PC と Server は Desktop / Services タブから図のとおり設定してください。

R1(config)# interface gigabitEthernet0/0 R1(config-if)# ip address 192.168.1.1 255.255.255.0 R1(config-if)# no shutdown R1(config-if)# exit R1(config)# interface gigabitEthernet0/1 R1(config-if)# ip address 192.168.2.1 255.255.255.0 R1(config-if)# no shutdown

設定できたら、PC-A と PC-B の両方から ping 192.168.2.100 が成功すること、PC-A の Web Browser で http://192.168.2.100 が開けることを確認します。ACL の学習は「全部通る状態」を先に確認してから始めるのが鉄則です。そうしないと、後で通信が止まったとき、ACL のせいなのか元々の設定ミスなのかを切り分けられなくなります。

STEP 2

番号付き標準 ACL を作る(まだ適用しない)

「PC-A からの通信だけを拒否し、それ以外は許可する」というルールを、番号 10 の標準 ACL として作ります。

R1(config)# access-list 10 deny host 192.168.1.10 R1(config)# access-list 10 permit any

コマンドの意味を1つずつ確認しましょう。access-list 10 は「番号 10 の ACL に1行追加する」という意味で、番号が 1〜99 の範囲なので Router は標準 ACL だと解釈します。deny host 192.168.1.10 は「送信元がこの1台のパケットを拒否」です。範囲で指定したい場合は access-list 10 deny 192.168.1.0 0.0.0.255 のように、サブネットマスクの 0 と 255 を反転させたワイルドカードマスクを使います。permit any は「残りはすべて許可」で、この行を忘れると暗黙の deny によって全通信が止まります。

ここで PC-A から Ping を打ってみてください。まだ全部通ります。ACL は作っただけでは何も起きず、インターフェースに適用して初めて効果が出ます。

STEP 3

G0/1 の out に適用して、止まる瞬間を見る

標準 ACL の定石どおり、宛先(Server)に近い G0/1 の out に適用します。ip access-group 10 out は「このインターフェースから出ていくパケットを ACL 10 で検査せよ」という意味です。

R1(config)# interface gigabitEthernet0/1 R1(config-if)# ip access-group 10 out R1(config-if)# end R1# show access-lists

PC-A から ping 192.168.2.100 を打つと、今度は Reply from 192.168.2.1: Destination host unreachable と表示されます。タイムアウトではなく R1 の IP から返事が来ているのがポイントで、「R1 が意図的に止めて、その旨を通知した」という証拠です。一方、PC-B からの Ping と、PC-A から PC-B への Ping は成功します。PC-A と PC-B の間の通信は R1 を通らないので、ACL の影響を受けません。

Ping を打った後に show access-lists をもう一度実行すると、deny の行に (4 match(es)) のようなヒット数が付きます。この数字が増える瞬間こそ、ACL が働いている証拠です。

STEP 4

【実験】わざと間違った方向に適用してみる

ここが今回のハイライトです。ACL 10 を G0/1 から外し、今度は LAN 側の G0/0 に out で適用します。「PC-A は LAN 側にいるのだから、LAN 側のインターフェースに付ければ止まるはず」という、初学者がやりがちな考え方をあえて再現します。

R1(config)# interface gigabitEthernet0/1 R1(config-if)# no ip access-group 10 out R1(config-if)# exit R1(config)# interface gigabitEthernet0/0 R1(config-if)# ip access-group 10 out

PC-A から Ping を打つ前に、結果を予想してみてください。実際に打つと——Ping は普通に成功してしまいます。

理由を追いかけましょう。G0/0 の out が検査するのは「R1 から LAN 側へ出ていく」パケット、つまり Server から PC-A へ戻ってくる帰りのパケットです。帰りのパケットの送信元は Server(192.168.2.100)であって PC-A ではないので、deny host 192.168.1.10 には一致せず、permit any で素通りします。show access-lists を見ても、deny 行のヒット数は増えていないはずです。ACL が見るのはパケットの中身であり、in / out は Router 視点の方向——この2つがつながると、ACL の動きは正確に予想できるようになります。

STEP 5

正しい方向 in に付け替えて、違いを確定させる

同じ G0/0 のまま、方向だけを in に変えます。今度は「LAN 側から R1 に入ってくる」パケット、つまり PC-A が送信元の行きのパケットが検査されます。

R1(config)# interface gigabitEthernet0/0 R1(config-if)# no ip access-group 10 out R1(config-if)# ip access-group 10 in

PC-A からの Ping は再び止まります。同じインターフェース・同じ ACL でも、方向が違うだけで結果が正反対になることを、自分の手で確認できました。「PC-A の通信を止める」なら、PC-A のパケットが通る G0/0 の in か G0/1 の out が正解で、G0/0 の out は不正解です。

実験が終わったら、次のステップに備えて no ip access-group 10 in で ACL を外し、全部通る状態に戻しておきます。

STEP 6

名前付き拡張 ACL で「Web だけ」止める

今度は「PC-A から Server への Web(HTTP)だけを拒否し、Ping を含む他の通信はすべて許可する」というルールを、名前付き拡張 ACL で作ります。

R1(config)# ip access-list extended BLOCK-A-WEB R1(config-ext-nacl)# deny tcp host 192.168.1.10 host 192.168.2.100 eq 80 R1(config-ext-nacl)# permit ip any any R1(config-ext-nacl)# exit R1(config)# interface gigabitEthernet0/0 R1(config-if)# ip access-group BLOCK-A-WEB in

ip access-list extended BLOCK-A-WEB で、名前付き拡張 ACL の編集モードに入ります(プロンプトが config-ext-nacl に変わります)。deny tcp host 192.168.1.10 host 192.168.2.100 eq 80 は「送信元 PC-A から、宛先 Server の TCP ポート 80(HTTP)へ向かうパケットだけを拒否」という意味です。標準 ACL では書けなかった宛先・プロトコル・ポートの条件が、1行に全部入っています。permit ip any any は「すべてのプロトコルの、すべての送信元から、すべての宛先への通信を許可」です。適用は、拡張 ACL の定石どおり送信元に近い G0/0 の in にします。

PC-A から ping 192.168.2.100 は成功するのに、Web Browser で http://192.168.2.100 を開こうとすると失敗します。PC-B からは両方成功します。「同じ相手なのに、通信の種類で通る・通らないが分かれる」——これが拡張 ACL の力です。

STEP 7

確認コマンドと、名前付き ACL ならではの編集

ACL の運用で使う確認コマンドは2つです。「どんなルールが何回ヒットしたか」は show access-lists、「どのインターフェースのどの方向に付いているか」は show ip interface で確認します。

R1# show access-lists R1# show ip interface gigabitEthernet0/0

show ip interface の出力にある Inbound access list is BLOCK-A-WEB という行が、適用場所と方向の証拠です。「ACL が効かない」トラブルの大半は方向の間違いなので、まずこのコマンドで事実を確認する習慣をつけてください。

さらに show access-lists を見ると、名前付き ACL の各行には 10、20 というシーケンス番号が付いています。名前付き ACL は、この番号を使って行単位の削除や挿入ができます。番号付き ACL では no access-list 10 とするとリスト全体が消えてしまうので、これは名前付きの大きな利点です。

R1(config)# ip access-list extended BLOCK-A-WEB R1(config-ext-nacl)# no 10 R1(config-ext-nacl)# 10 deny tcp host 192.168.1.10 host 192.168.2.100 eq 80 R1(config-ext-nacl)# exit R1(config)# exit R1# copy running-config startup-config

no 10 で 10 番の行だけを削除し、10 deny ... のように行頭に番号を付けて打つと、その位置に行を挿入できます。最後に設定を保存して完了です。

ここで一度立ち止まって考える

STEP 4 で G0/0 の out に適用したとき、なぜ Ping が通ってしまったのかを、自分の言葉で説明してみてください。「G0/0 の out を通るのは帰りのパケットで、その送信元は Server だから deny に一致しない」——ここまで言えれば、in と out はもう混同しません。

Three Rules

ACLを支配する3つのルール

1. 上から順、最初の一致で即決

パケットはリストを上から照合され、最初に一致した行で permit / deny が確定します。それより下の行は見られません。permit any を先頭に書くと、以降の deny は永遠に評価されない「死んだ行」になります。

2. 末尾には暗黙の deny

どの行にも一致しなかったパケットは、設定画面には表示されない「暗黙の deny」ですべて破棄されます。deny だけの ACL を適用すると全通信が止まるのはこのためです。許可行とセットで書くのが基本です。

3. 1インターフェース・1方向に1つ

1つのインターフェースの同じ方向に適用できる ACL は1つだけです。同じ方向に別の ACL を適用すると、前の ACL は上書きされて外れます。条件を増やしたいときは、同じ ACL に行を足します。

If It Fails

うまくいかないときの確認ポイント

通信が全部止まった

ほぼ確実に permit any(拡張なら permit ip any any)の入れ忘れです。R1# show access-lists で許可行があるかを確認します。番号付き ACL は行の削除ができないので、no access-list 10 で消してから作り直します。

ACL が効いていない

まず R1# show ip interface gigabitEthernet0/0 で、意図したインターフェースの意図した方向に付いているかを確認します。STEP 4 の実験のとおり、方向が違うと ACL は「存在するのに一度も評価されない」状態になります。ヒット数が増えない ACL は、パケットの通り道にいません。

止めたいものと違う通信が止まる

行の順番を疑います。ACL は最初の一致で即決なので、広い条件の行(permit any など)が狭い条件の deny より上にあると、deny は機能しません。show access-lists でシーケンス番号順に読み直し、名前付きなら行の挿入で順番を直します。
