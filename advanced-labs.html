DHCP / NAT / ACL / Capstone Challenge

自宅のルーターがやっていることを、自分の手で作る。

ここまでの単元は「届くようにする」ための学習でした。この単元では、IP を自動で配る DHCP、社内のプライベート IP をインターネット向けに変換する NAT、通信を制御する ACL という、実務のネットワークに必ず登場する3つのサービスを、1つのトポロジに順番に積み上げます。最後には、全単元の知識を総動員する総合チャレンジも用意しました。

Unit Topology

この単元で構築する構成

社内 LAN と「インターネット側」を R1 が仲介する構成です。ISP Router の先にある Server を、インターネット上の Web サーバーに見立てます。Lab A から Lab C まで、この1つの構成を育てていきます。

PC-A / PC-BEnd DeviceIPはDHCPで自動取得
192.168.10.0 /24
SW1Catalyst 2960社内LANのスイッチ
R1社内Router (1941)G0/0: 192.168.10.1
G0/1: 203.0.113.1 /30
ISPISR 1941G0/0: 203.0.113.2 /30
G0/1: 198.51.100.1 /24
ServerWeb Server198.51.100.10 /24
GW 198.51.100.1

アドレス設計の意図

  • 192.168.10.0 /24 は社内専用のプライベート IP です
  • 203.0.113.0 /30 と 198.51.100.0 /24 をグローバル IP(インターネット側)に見立てます
  • プライベート IP のままではインターネット側と通信できない、という前提が NAT の出発点です

先に手を動かす準備

  • R1・ISP の各インターフェースに図のとおり IP を設定し、no shutdown まで済ませます
  • Server には固定で 198.51.100.10 /24、GW 198.51.100.1 を設定します
  • PC の IP はまだ設定しません。Lab A で DHCP から配ります

Packet Tracer で選ぶ Router は 1941

R1・ISP の2台とも、GigabitEthernet ポート(G0/0・G0/1)を持つ 1941(2911 でも可)を選んでください。1841 のような FastEthernet 搭載機種を選んだ場合は、interface gigabitEthernet0/0 を interface fastEthernet0/0 のようにポート名を読み替える必要があります(NAT の ip nat inside source list 1 interface ... のポート名も同様です)。実際のポート名は show ip interface brief で確認できます。詳しくはPacket Tracer の使い方の「装置の選び方とポート名の関係」を参照してください。

Lab A / DHCP

Lab A: IPアドレスを自動で配る

ここまでの単元では PC に IP を手入力してきましたが、実際のオフィスで社員の PC に1台ずつ手入力することはありません。R1 を DHCP サーバーにして、IP・サブネットマスク・ゲートウェイ・DNS を自動配布します。

STEP 1

配布しない IP を予約する

Router や Server 用の IP が PC に配られてしまわないように、先頭の範囲を配布対象から外します。

R1(config)# ip dhcp excluded-address 192.168.10.1 192.168.10.9
STEP 2

DHCP プールを作る

「どのネットワークの IP を、どんな付属情報と一緒に配るか」をプールとして定義します。default-router が PC のデフォルトゲートウェイに、dns-server が PC の DNS 設定になります。

R1(config)# ip dhcp pool LAN-POOL R1(dhcp-config)# network 192.168.10.0 255.255.255.0 R1(dhcp-config)# default-router 192.168.10.1 R1(dhcp-config)# dns-server 198.51.100.10 R1(dhcp-config)# exit
STEP 3

PC 側を DHCP に切り替えて確認する

PC-A と PC-B の Desktop → IP Configuration で「DHCP」を選びます。192.168.10.10 以降の IP が自動で入れば成功です。Router 側でも、誰に何を配ったかを確認できます。

R1# show ip dhcp binding
Lab B / NAT

Lab B: プライベートIPを変換して外へ出す

この時点で PC から Server へ Ping を打っても、返事は戻ってきません。インターネット側(ISP)はプライベート IP 192.168.10.0/24 への戻り道を知らないからです。そこで R1 に NAT(PAT)を設定し、社内の全 PC が R1 の外側 IP 203.0.113.1 を共有して外に出られるようにします。これは家庭のブロードバンドルーターと同じ仕組みです。

STEP 1

変換対象を ACL で指定する

「どの送信元 IP を変換するか」を、ACL 入門で学んだ番号付き標準 ACL で定義します。ワイルドカードマスクもここで再登場します。

R1(config)# access-list 1 permit 192.168.10.0 0.0.0.255
STEP 2

NAT の内側と外側を宣言する

NAT では「どちらが社内(inside)で、どちらがインターネット側(outside)か」を Router に教える必要があります。

R1(config)# interface gigabitEthernet0/0 R1(config-if)# ip nat inside R1(config-if)# exit R1(config)# interface gigabitEthernet0/1 R1(config-if)# ip nat outside R1(config-if)# exit
STEP 3

PAT と外向きの経路を設定する

overload が PAT(1つのグローバル IP を複数 PC で共有)の指定です。あわせて、外向きの通信をすべて ISP へ送るデフォルトルートも入れます。

R1(config)# ip nat inside source list 1 interface gigabitEthernet0/1 overload R1(config)# ip route 0.0.0.0 0.0.0.0 203.0.113.2
STEP 4

変換の様子を観察する

PC-A から 198.51.100.10 へ Ping を打ち、続けて PC-A の Web Browser で http://198.51.100.10 を開いてみてください。そのうえで R1 の変換テーブルを見ると、送信元が 203.0.113.1 に書き換えられている様子が確認できます。

R1# show ip nat translations

ここで一度立ち止まって考える

ISP Router には 192.168.10.0/24 への経路を1本も書いていないのに、なぜ Ping の返事が PC まで戻れるのでしょうか。答えは「ISP から見ると、通信相手は R1 の 203.0.113.1 に見えているから」です。この理屈が自分の言葉で説明できれば、NAT は理解できています。

Lab C / ACL

Lab C: 通信を選んで止める

最後は、ACL 入門で学んだアクセスコントロールリストの実践編です。「PC-B からは Server への Web アクセスを禁止するが、それ以外の通信は許可する」というルールを、NAT が動いている R1 に設定します。全部止めるのではなく、狙った通信だけを止めるのがポイントです。

STEP 1

PC-B の IP を固定する

DHCP のままだと PC-B の IP が変わる可能性があるため、この Lab では PC-B に 192.168.10.20 /24(GW 192.168.10.1)を手動設定して対象を固定します。

STEP 2

拡張 ACL でルールを書く

拡張 ACL は「誰が・どこへ・何の通信か」まで指定できます。1行目で PC-B から Server への Web(HTTP)だけを拒否し、2行目でそれ以外をすべて許可します。

R1(config)# access-list 100 deny tcp host 192.168.10.20 host 198.51.100.10 eq 80 R1(config)# access-list 100 permit ip any any

2行目の permit ip any any を忘れると、ACL の末尾に隠れている「暗黙の deny」によって、社内からの通信がすべて止まってしまいます。ACL 事故の定番なので、必ずセットで覚えてください。

STEP 3

インターフェースに適用する

ACL は作っただけでは動きません。「どのインターフェースの、どの向きの通信に適用するか」を指定して初めて効果が出ます。今回は、拡張 ACL の定石どおり送信元に近い側、つまり社内 LAN から R1 に入ってくる通信(in)に適用します。in と out で迷ったら、ACL 入門の「わざと間違える実験」を思い出してください。

R1(config)# interface gigabitEthernet0/0 R1(config-if)# ip access-group 100 in R1(config-if)# exit
STEP 4

狙いどおりに効いているか確認する

PC-B の Web Browser から http://198.51.100.10 が開けなくなり、PC-B からの Ping と、PC-A からの Web アクセスは今までどおり成功すれば合格です。ACL の各行に何回ヒットしたかも確認できます。

R1# show access-lists
Capstone Challenge

総合チャレンジ: 小さなオフィスネットワークを一人で作る

手順書はここにはありません。要件だけを見て、これまでの全単元の知識を組み合わせて構築してみてください。詰まったら、各単元のページとトラブルシュートの「確認の順番」に戻れば必ずヒントがあります。

要件

  • 社内に VLAN 10(営業部)と VLAN 20(開発部)を作り、L3 Switch の SVI で VLAN 間通信させる(07 の内容)
  • L3 Switch と R1、R1 と ISP を /30 のリンクでつなぎ、社内区間は OSPF で経路交換する(10 の内容)
  • 各 VLAN の PC には R1 から DHCP で IP を配る(この単元の Lab A の発展。プールが2つ必要です)
  • R1 の NAT(PAT)経由で、全 PC から ISP 先の Server に届くようにする(Lab B の内容)
  • VLAN 20 から Server への Web アクセスだけを ACL で禁止する(Lab C の内容)

合格チェックリスト

  • VLAN 10 の PC から VLAN 20 の PC へ Ping が通る
  • すべての PC が DHCP で正しいゲートウェイを取得している
  • R1 の経路表に OSPF で学んだ VLAN のネットワークが O 付きで載っている
  • VLAN 10 の PC から Server の Web ページが開ける
  • VLAN 20 の PC は Web だけ失敗し、Ping は成功する

ここまで来たあなたへ

このチャレンジが自力で完成すれば、CCNA の学習範囲のうち、構築の中核部分にはすでに手が届いています。うまくいかない箇所こそ最大の教材です。Simulation モードでパケットを追いながら、どの単元の知識が抜けていたのかを言葉にしてみてください。
